## Context

### Current State

The MPS (Media Processing Service) operation resources in the Terraform provider call Tencent Cloud APIs that return `TaskId` in their responses. However, these task IDs are used internally as the Terraform resource ID (`d.SetId(taskId)`) but are not exposed as readable computed fields in the schema.

Affected resources include:
- `tencentcloud_mps_process_media_operation` - calls ProcessMedia API
- `tencentcloud_mps_edit_media_operation` - calls EditMedia API
- `tencentcloud_mps_process_live_stream_operation` - calls ProcessLiveStream API
- Potentially other MPS operation resources

All these APIs follow the same pattern:
1. User provides media processing parameters
2. API is called synchronously and returns immediately with a `TaskId`
3. TaskId can be used to query task status asynchronously
4. Resource ID in Terraform state is set to the TaskId

### Constraints

- **Backward Compatibility**: Cannot break existing Terraform configurations or state
- **Schema Stability**: Cannot modify or remove existing schema fields
- **Minimal Changes**: This is a simple additive change, should not introduce complex logic
- **Vendor SDK**: Must use existing vendor directory SDK (no SDK upgrades required)

## Goals / Non-Goals

**Goals:**
- Expose `task_id` as a computed field in affected MPS operation resources
- Enable users to reference task IDs from these resources in other Terraform resources
- Provide visibility into which task ID was generated by each operation
- Maintain full backward compatibility with existing configurations

**Non-Goals:**
- Changing the behavior or logic of existing operations
- Adding new APIs or SDK calls
- Modifying how resource IDs are set or used internally
- Adding task status querying capabilities (out of scope)

## Decisions

### 1. Field Definition Style

**Decision**: Add `task_id` as a computed field (not optional or required)

**Rationale**:
- TaskId is returned by the API, not provided by the user
- Computed fields are appropriate for API-generated identifiers
- Aligns with existing patterns in the codebase (e.g., `task_id` in `data_source_tc_mps_tasks.go`)
- Users cannot influence the TaskId value, it's purely a read-only output

**Alternative Considered**: Make `task_id` optional

**Rejected**: Users don't provide TaskId - it's generated by the API. Optional fields would imply user input capability.

### 2. Field Placement and Naming

**Decision**: Add `task_id` at the top level of each resource schema using snake_case

**Rationale**:
- Consistent with existing computed fields like `id`, `create_time`
- Follows Terraform naming conventions
- Easy to reference in other resources using `resource_name.task_id`
- Matches the SDK field name (`TaskId` in Go, `task_id` in JSON/HCL)

**Alternative Considered**: Use `taskId` (camelCase) or nest under a different structure

**Rejected**: Terraform conventions favor snake_case for field names. Nesting would complicate reference patterns.

### 3. Setting the Field Value

**Decision**: Set `task_id` in the Create function immediately after API call succeeds

**Rationale**:
- TaskId is available immediately from `response.Response.TaskId`
- Resource ID is already set to the same value: `d.SetId(taskId)`
- Setting computed field at the same time is efficient and clear
- No need to fetch additional data from the API

**Code Pattern**:
```go
// After successful API call
taskId = *response.Response.TaskId
d.SetId(taskId)
// Add this line
d.Set("task_id", taskId)
```

**Alternative Considered**: Set `task_id` in the Read function

**Rejected**: Read function for operation resources is often a no-op (just returns nil). Setting in Create is more direct and reliable.

### 4. Read Function Behavior

**Decision**: Keep Read function as-is (typically no-op for operation resources)

**Rationale**:
- Operation resources are trigger-style resources that perform one-time actions
- They don't represent persistent state that needs periodic refresh
- Setting `task_id` in Create is sufficient
- Avoids unnecessary API calls that would fail or return outdated information

**Alternative Considered**: Fetch task status in Read function

**Rejected**: Out of scope. Would require calling DescribeTask APIs and handling task lifecycle, which is a separate feature.

### 5. Testing Strategy

**Decision**: Verify field is set correctly in acceptance tests

**Rationale**:
- Computed fields need test coverage to ensure they're populated
- Existing acceptance tests should already verify Create succeeds
- Add simple assertions that `task_id` is not empty and matches resource ID

**Test Pattern**:
```go
// After resource creation
resource.TestCheckResourceAttr(resourceName, "task_id", rs.Primary.ID),
```

## Risks / Trade-offs

### Risk: TaskId Already Used as Resource ID

**Risk**: We're adding `task_id` as a computed field when it's already being used as the resource ID (`d.SetId(taskId)`). This creates redundancy.

**Mitigation**:
- This is intentional and provides user convenience
- Users can now reference `resource.task_id` instead of needing to understand it's the same as `resource.id`
- Document that `task_id` and `id` are the same value for clarity
- No functional impact - just adds a convenient alias

### Risk: Inconsistent Implementation Across Resources

**Risk**: Not all MPS operation resources may return TaskId, leading to incomplete implementation.

**Mitigation**:
- Research each affected resource's API response before implementation
- Only add `task_id` to resources where the API actually returns it
- Verify with vendor SDK models (e.g., `ProcessMediaResponseParams` has `TaskId *string`)
- Test each resource individually

### Risk: Breaking Existing State

**Risk**: Adding a new computed field could theoretically affect Terraform state handling.

**Mitigation**:
- Computed fields are additive and backward compatible
- Terraform automatically adds new computed fields to state on next refresh
- No force-new or schema changes that trigger recreation
- Test with existing state to ensure smooth migration

## Migration Plan

### Deployment Steps

1. **Implementation**: Add `task_id` computed field to each affected resource
2. **Testing**: Run acceptance tests for all modified resources
3. **Documentation**: Update resource documentation files
4. **Code Review**: Ensure consistent implementation across all resources

### Rollback Strategy

- If issues arise, simply remove the `task_id` field addition
- This is a pure additive change with no complex dependencies
- No data migration or state changes required
- Existing configurations continue to work without the field

### No Database or State Migration Required

- Terraform state will automatically include the new computed field on next apply
- No manual state manipulation needed
- Users benefit from the new field automatically
