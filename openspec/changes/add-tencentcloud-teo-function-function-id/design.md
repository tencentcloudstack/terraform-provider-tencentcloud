## Context

The `tencentcloud_teo_function` resource currently supports creating edge functions in TencentCloud EdgeOne (TEO) service. The resource has a `function_id` field that is marked as `Computed: true`, meaning it's read-only and automatically generated by the CreateFunction API response. Users cannot specify a custom FunctionId during resource creation.

The current implementation:
- Schema defines `function_id` as Computed only
- CreateFunction API does not accept FunctionId in the request
- FunctionId is read from the API response and stored in the state
- Resource ID is a composite of `zone_id` and `function_id`

This limitation prevents users from:
- Specifying a custom FunctionId when creating a new function
- Importing existing functions with known FunctionIds
- Migrating functions from other systems with pre-defined IDs

## Goals / Non-Goals

**Goals:**
- Enable users to optionally specify `function_id` when creating a `tencentcloud_teo_function` resource
- Maintain full backward compatibility with existing configurations
- Support both scenarios: API-generated FunctionId (default) and user-specified FunctionId
- Update tests to cover both creation scenarios
- Update documentation to reflect the new optional parameter

**Non-Goals:**
- Changing the resource ID structure (remains `zone_id#function_id`)
- Modifying the Update or Delete operations
- Adding validation rules for FunctionId format (will use API validation)
- Changing other fields in the resource schema

## Decisions

### 1. Schema Modification Strategy

**Decision:** Change `function_id` from `Computed: true` to `Optional: true, Computed: true`

**Rationale:**
- `Optional: true` allows users to specify the field during creation
- `Computed: true` ensures the field is always present in the state, even when not specified by the user
- This combination is the standard pattern for optional identifiers in Terraform providers
- Maintains backward compatibility: existing resources without function_id will work as before

**Alternative Considered:** Keep `Computed: true` and add a separate custom_id field
- Rejected: Adds unnecessary complexity and duplicates the concept
- Would require mapping logic between custom_id and function_id

### 2. API Request Handling

**Decision:** Conditionally include FunctionId in CreateFunction request based on user input

**Rationale:**
- The TeoV20220901 SDK's CreateFunctionRequest likely accepts an optional FunctionId parameter
- Only set the field when the user provides a value
- When not provided, the API will generate a FunctionId as before
- This approach minimizes changes and maintains the existing flow

**Implementation:**
```go
if v, ok := d.GetOk("function_id"); ok {
    request.FunctionId = helper.String(v.(string))
}
```

### 3. ID Construction and Import

**Decision:** Keep the composite ID structure unchanged (`zone_id#function_id`)

**Rationale:**
- Maintains backward compatibility with existing state
- No changes needed to the Importer (schema.ImportStatePassthrough)
- Users can import existing functions by specifying `terraform import tencentcloud_teo_function.example zone_id#function_id`

**Alternative Considered:** Change to single `function_id` as the resource ID
- Rejected: Would break existing state and require migration
- zone_id is still required for API operations

### 4. Read Operation Consistency

**Decision:** No changes to Read operation

**Rationale:**
- The Read operation already reads `function_id` from the API response (line 180-183)
- It sets both the schema field and the local variable
- This works correctly for both scenarios (user-specified or API-generated)
- The read operation is idempotent and doesn't need modification

## Risks / Trade-offs

### Risk 1: API Compatibility Issue
**Risk:** The CreateFunction API might not accept an optional FunctionId parameter, causing errors when users specify it

**Mitigation:**
- Verify the API supports optional FunctionId by checking the SDK definition
- If the API doesn't support it, this design will need to be revisited
- Consider checking API documentation or testing with the SDK first

**Current Understanding:** Based on common patterns in TencentCloud APIs, CreateFunction likely accepts an optional FunctionId parameter, but this should be verified.

### Risk 2: State Migration
**Risk:** Existing resources with computed function_id in state might need migration

**Mitigation:**
- Since we're keeping `Computed: true`, existing state will continue to work
- The function_id is already stored in state from the API response
- No state migration should be required

### Risk 3: Validation Conflicts
**Risk:** User might specify an invalid FunctionId format that the API rejects

**Mitigation:**
- Let the API handle validation - return API errors to the user
- Do not add custom validation logic in the provider
- The API will return descriptive error messages for invalid FunctionId

### Trade-off: Complexity vs. Flexibility
**Trade-off:** Adding optional FunctionId increases schema complexity slightly but provides much-needed flexibility for users

**Justification:** The complexity is minimal (one field attribute change) and the benefit is significant (supports import and custom IDs). This is a common pattern in Terraform providers.

## Migration Plan

### Deployment Steps
1. Modify the schema definition in `resource_tc_teo_function.go`
2. Update the CreateFunction request to include optional FunctionId
3. Update test cases to cover both scenarios (with and without function_id)
4. Update documentation in `resource_tc_teo_function.md`
5. Run existing tests to ensure backward compatibility

### Rollback Strategy
- Revert the schema change and code modifications
- This is a feature addition, so rollback is straightforward
- Existing resources are not affected due to backward compatibility

## Open Questions

1. **API Support Verification:** Does the TeoV20220901 CreateFunction API actually support an optional FunctionId parameter?
   - This should be verified by checking the SDK source code or API documentation
   - If not supported, the design will need to be adjusted

   - Likely returns an error (duplicate ID)
   - Need to document this behavior in the provider documentation

3. **FunctionId Format:** Are there any format restrictions on FunctionId (length, allowed characters)?

   - No need to implement custom validation if the API enforces rules
