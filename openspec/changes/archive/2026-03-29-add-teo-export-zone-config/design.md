## Context

Terraform Provider for TencentCloud 当前已经支持 TEO (TencentCloud EdgeOne) 服务的多个资源，包括站点管理、规则引擎、缓存配置等。然而，目前缺少导出站点完整配置的数据源能力，这导致用户在需要备份、迁移或审计站点配置时面临困难。

TEO 站点配置包含多个维度的信息，包括基础配置、加速域名配置、安全策略、缓存规则等。用户需要通过一个数据源能够查询到站点的完整配置信息。

## Goals / Non-Goals

**Goals:**
- 提供一个新的数据源 `tencentcloud_teo_export_zone_config`，支持导出 TEO 站点的完整配置
- 数据源返回的配置应包含站点的所有关键信息，包括但不限于：站点 ID、站点名称、状态、区域配置、域名列表、安全策略、缓存规则等
- 确保数据源的性能和可靠性，支持大规模配置查询
- 提供完整的文档和测试用例

**Non-Goals:**
- 不修改现有的 TEO 资源或数据源
- 不实现配置的导入功能（导入已有 `tencentcloud_teo_zone_config` 资源）
- 不涉及站点配置的在线编辑或管理功能

## Decisions

### 1. 数据源 Schema 设计
- **决策**: 使用 `zone_id` 作为必填输入参数
- **理由**: 站点 ID 是查询配置的唯一标识符，符合 TEO API 的调用习惯
- **替代方案考虑**: 使用站点名称作为查询参数，但站点名称可能重复，不如 ID 唯一

### 2. API 调用策略
- **决策**: 通过 TEO SDK 的 `DescribeZoneConfig` 或类似 API 获取配置
- **理由**: 使用官方提供的 API，确保数据的准确性和完整性
- **替代方案考虑**: 通过组合多个 API 获取配置，但会增加复杂性和不一致风险

### 3. 错误处理
- **决策**: 遵循 Provider 的错误处理模式，使用 `defer tccommon.LogElapsed()` 和 `defer tccommon.InconsistentCheck()`
- **理由**: 与现有代码保持一致，提供统一的错误处理和日志记录
- **替代方案考虑**: 自定义错误处理逻辑，但会增加维护成本

### 4. 重试机制
- **决策**: 对于可能存在最终一致性的 API 调用，使用 `helper.Retry()` 进行重试
- **理由**: TEO 作为分布式服务，可能存在短暂的一致性问题，重试机制可以提高可靠性
- **替代方案考虑**: 不使用重试，但可能导致偶发失败

### 5. 性能优化
- **决策**: 对于大型站点配置，考虑使用分页查询或增量查询
- **理由**: 避免一次性加载过大的配置导致超时或内存问题
- **替代方案考虑**: 一次性加载所有配置，但对于复杂站点可能导致性能问题

## Risks / Trade-offs

- **风险**: TEO API 可能会变更，导致数据源需要维护
  - **缓解**: 使用版本化的 SDK，定期更新并监控 API 变更

- **风险**: 大型站点配置可能导致查询超时
  - **缓解**: 添加合理的超时设置，使用重试机制，并在文档中说明

- **风险**: 配置信息可能包含敏感信息
  - **缓解**: 在文档中明确说明用户需要妥善保护导出的配置信息

- **权衡**: 配置的完整性 vs 查询性能
  - **决策**: 优先保证配置的完整性，通过优化 API 调用和缓存来提升性能

## Migration Plan

由于这是新增数据源，不涉及现有资源的迁移：
1. 实现数据源代码
2. 编写和运行测试用例
3. 编写文档和示例
4. 通过 CI/CD 流程验证
5. 发布新版本

如需回滚，可以：
- 通过 Provider 版本管理，用户可以降级到不包含此数据源的版本
- 数据源不影响现有资源的状态，回滚无副作用
