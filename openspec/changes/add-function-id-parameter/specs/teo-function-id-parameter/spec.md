## ADDED Requirements

### Requirement: User can specify FunctionId when creating Teo function
The tencentcloud_teo_function resource SHALL allow users to specify the FunctionId parameter when creating a new function. The FunctionId parameter SHALL be marked as both Optional and Computed in the resource schema.

#### Scenario: User creates function with specified FunctionId
- **WHEN** user creates a tencentcloud_teo_function resource with a specified function_id value
- **THEN** the CreateFunction API call SHALL include the provided FunctionId parameter
- **AND** the response FunctionId SHALL match the user-specified value
- **AND** the resource state SHALL store the FunctionId

#### Scenario: User creates function without specifying FunctionId
- **WHEN** user creates a tencentcloud_teo_function resource without specifying function_id
- **THEN** the CreateFunction API call SHALL NOT include a FunctionId parameter
- **AND** the API response SHALL generate and return a FunctionId
- **AND** the resource state SHALL store the auto-generated FunctionId
- **AND** the function_id attribute SHALL be populated with the server-generated value

### Requirement: FunctionId parameter supports read operations
The tencentcloud_teo_function resource SHALL support reading the FunctionId from the existing function instance regardless of whether it was user-specified or auto-generated.

#### Scenario: Read existing function with user-specified FunctionId
- **WHEN** user reads a tencentcloud_teo_function resource that was created with a user-specified function_id
- **THEN** the resource SHALL return the FunctionId that was specified during creation

#### Scenario: Read existing function with auto-generated FunctionId
- **WHEN** user reads a tencentcloud_teo_function resource that was created without specifying function_id
- **THEN** the resource SHALL return the FunctionId that was auto-generated by the server

### Requirement: FunctionId parameter maintains backward compatibility
The modification to the function_id parameter SHALL NOT break existing Terraform configurations or states that rely on the Computed-only behavior.

#### Scenario: Existing configuration continues to work
- **WHEN** user applies an existing Terraform configuration that does not specify function_id
- **THEN** the configuration SHALL continue to work without modification
- **AND** the function_id attribute SHALL be populated with the server-generated value

#### Scenario: Existing state continues to work
- **WHEN** user runs Terraform plan/apply on an existing state with function_id already set
- **THEN** the operation SHALL succeed without requiring state modification
