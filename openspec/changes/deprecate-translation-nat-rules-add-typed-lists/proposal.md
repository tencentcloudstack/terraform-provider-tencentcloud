# Proposal: Deprecate `translation_nat_rules` and add three typed ordered list fields

## Why

NAT translation rules naturally split into three buckets — `LOCAL/NETWORK_LAYER`, `LOCAL/TRANSPORT_LAYER`, `PEER/NETWORK_LAYER` — each with its own primary-key uniqueness contract:

* Three-layer (NETWORK_LAYER) buckets: `original_ip` is unique within the bucket.
* Four-layer (TRANSPORT_LAYER) bucket: `translation_ip` is unique within the bucket.

Two prior approaches both failed:

1. **Original `translation_nat_rules` (TypeSet, default hash)** — hashed all five fields, so editing description forced DELETE+CREATE. No Modify path was possible.
2. **TypeSet with custom hash on the primary key** — Solved the Modify path on paper, but the SDK's Set machinery type-asserts inner map values to `string`. When Read populates the map with `*string` (the SDK's struct field type), the assertions silently fail and every rule collapses into one bucket. Even after fixing dereferencing, plan output renders set elements in unstable hash order, which is hard for users to read when there are many rules.

The product team's clarification is decisive: rules are an unordered set, but **users still want to see them in their own HCL order in plan output**. The clean solution is:

* Use **three separate `TypeList` fields** so each is naturally ordered.
* Solve the position-drift problem (where API return order ≠ user HCL order) at Read time by re-aligning the Read output to the order the user wrote in HCL.

## What Changes

* **The three new typed lists become the primary schema.** Documentation, examples, and the migration note all steer users to them.
* **Mark `translation_nat_rules` as `Deprecated` (NOT removed).** Keep functional so existing `.tf` files plan/apply unchanged.
* **Add three new `TypeList` fields**, one per bucket, with bucket-tailored inner schema:
  * `local_network_layer_rules` — `translation_ip`, `original_ip`, `description`
  * `local_transport_layer_rules` — `translation_ip`, `description` (no `original_ip`)
  * `peer_network_layer_rules` — `translation_ip`, `original_ip`, `description`
* **Pairwise `ConflictsWith`** between legacy and any new field; the three new fields are NOT mutually exclusive (users can mix them).
* **Read aligns the typed lists to user-HCL order** by inspecting `d.Get(field)` (which holds the user-config order during Refresh after plan/apply) and reordering the API response to match. Rules present in the API response but absent from user config are appended in API order.
* **Update**: legacy `translation_nat_rules` branch unchanged; three new branches each diff by primary key and emit DELETE → MODIFY → CREATE in that order.
* **Inner `translation_ip` / `original_ip` are NOT ForceNew.** Editing the non-key IP becomes an in-place change routed to MODIFY; editing the key IP is naturally split into DELETE+CREATE by the diff algorithm.

## Impact

* **Affected resources**: `tencentcloud_vpc_private_nat_gateway_translation_nat_rule`
* **Affected specs**: new capability `vpc-private-nat-gateway-translation-nat-rule-typed-lists`
* **Affected files**:
  * `tencentcloud/services/vpc/resource_tc_vpc_private_nat_gateway_translation_nat_rule.go`
  * `tencentcloud/services/vpc/resource_tc_vpc_private_nat_gateway_translation_nat_rule.md`
  * `website/docs/r/vpc_private_nat_gateway_translation_nat_rule.html.markdown` (regenerated by `make doc`)
* **State compatibility**: `translation_nat_rules` schema is unchanged for legacy users; no `SchemaVersion` upgrade is needed. Users who migrate to the typed lists can `terraform refresh` to populate the new fields with no API write.
