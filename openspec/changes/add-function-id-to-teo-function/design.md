# Design Document: Add FunctionId Parameter to tencentcloud_teo_function

## Architecture Overview

This change modifies the existing `tencentcloud_teo_function` resource by changing the `function_id` parameter from Computed-only to Computed+Optional. This allows users to specify an existing function ID during creation, which is then passed to the CreateFunction API.

### Create Flow (With FunctionId)

```

    │

    ├─ name = "my-function"
    ├─ function_id = "func-xxx"  ← NEW: Optional input
    ├─ content = "..."
    └─ remark = "..."
    │
    ▼
Schema Validation
    │
    ├─ zone_id is required
    ├─ name is required
    ├─ function_id is optional (NEW)
    ├─ content is required
    └─ remark is optional
    │
    ▼
resourceTencentCloudTeoFunctionCreate()
    │
    ├─ Extract zone_id
    ├─ Extract name
    ├─ Extract function_id (NEW) if provided
    ├─ Extract content
    ├─ Extract remark
    │
    ▼
TeoService.DescribeTeoFunctionsById() or CreateFunction()
    │
    ├─ If function_id is provided (NEW):
    │   ├─ Check if function already exists
    │   ├─ If exists: Use existing function ID
    │   └─ If not exists: Create new function with function_id
    │   │
    │   └─ Build CreateFunctionRequest with function_id
    │
    └─ If function_id is NOT provided:
        └─ Build CreateFunctionRequest WITHOUT function_id

    │
    ▼

    │
    └─ Function ID returned
    │
    ▼
resourceTencentCloudTeoFunctionRead()
    │
    └─ Read and populate state (see Read Flow)
```

## Schema Design

### Modified function_id Parameter

```go
"function_id": {
    Type:        schema.TypeString,
    Optional:    true,    // CHANGED from Computed-only
    Computed:    true,    // Retain Computed attribute
    Description: "ID of the Function. If not specified, the API will generate one automatically.",
}
```

**Key Changes:**
- **Before**: `Computed: true` (read-only, auto-generated by API)
- **After**: `Optional: true, Computed: true` (can be specified by user OR auto-generated)
- **ForceNew**: Not set (function_id can be provided during creation but doesn't force recreation)

## Service Layer Signature

### Modified Create Function Handling

```go
func resourceTencentCloudTeoFunctionCreate(d *schema.ResourceData, meta interface{}) error {
    defer tccommon.LogElapsed("resource.tencentcloud_teo_function.create")()
    defer tccommon.InconsistentCheck(d, meta)()

    logId := tccommon.GetLogId(tccommon.ContextNil)

    ctx := tccommon.NewResourceLifeCycleHandleFuncContext(context.Background(), logId, d, meta)

    var (
        zoneId     string
        functionId string
    )
    var (
        request  = teov20220901.NewCreateFunctionRequest()
        response = teov20220901.NewCreateFunctionResponse()
    )

    // Extract zone_id
    if v, ok := d.GetOk("zone_id"); ok {
        zoneId = v.(string)
    }
    request.ZoneId = helper.String(zoneId)

    // Extract function_id (NEW)
    if v, ok := d.GetOk("function_id"); ok {
        functionId = v.(string)
        request.FunctionId = helper.String(functionId)
    }

    // Extract existing parameters (unchanged)
    if v, ok := d.GetOk("name"); ok {
        request.Name = helper.String(v.(string))
    }

    if v, ok := d.GetOk("remark"); ok {
        request.Remark = helper.String(v.(string))
    }

    if v, ok := d.GetOk("content"); ok {
        request.Content = helper.String(v.(string))
    }

    // Execute CreateFunction request
    err := resource.Retry(tccommon.WriteRetryTimeout, func() *resource.RetryError {
        result, e := meta.(tccommon.ProviderMeta).GetAPIV3Conn().UseTeoV20220901Client().CreateFunctionWithContext(ctx, request)
        if e != nil {
            return tccommon.RetryError(e)
        } else {
            log.Printf("[DEBUG]%s api[%s] success, request body [%s], response body [%s]\n", logId, request.GetAction(), request.ToJsonString(), result.ToJsonString())
        }
        response = result
        return nil
    })
    if err != nil {
        log.Printf("[CRITAL]%s create teo function failed, reason:%+v", logId, err)
        return err
    }

    // Get function_id from response
    functionId = *response.Response.FunctionId

    // Wait for function to be active
    if _, err := (&resource.StateChangeConf{
        Delay:      10 * time.Second,
        MinTimeout: 3 * time.Second,
        Pending:    []string{"false"},
        Refresh:    resourceTeoFunctionCreateStateRefreshFunc_0_0(ctx, zoneId, functionId),
        Target:     []string{"true"},
        Timeout:    600 * time.Second,
    }).WaitForStateContext(ctx); err != nil {
        return err
    }

    // Set resource ID
    d.SetId(strings.Join([]string{zoneId, functionId}, tccommon.FILED_SP))

    return resourceTencentCloudTeoFunctionRead(d, meta)
}
```

## Update Strategy

### No Changes Required for Update

The `function_id` parameter is not updatable through the Update operation because:
1. Function ID is typically immutable after creation
2. This parameter is only relevant during the Create operation
3. If a user wants to change the function ID, they should recreate the resource

**Note**: The Update operation (`resourceTencentCloudTeoFunctionUpdate`) remains unchanged.

## Error Handling

### Validation Errors

```go
// No additional validation needed for function_id
// The API will handle validation of function_id format and uniqueness
```

### API Error Handling

Common error scenarios:
1. **Duplicate FunctionId**: If a user provides a function_id that already exists
   - API returns error (e.g., "FunctionId already exists")
   - Terraform displays API error message
   - User provides a different function_id or omits it

2. **Invalid FunctionId Format**: If the provided function_id doesn't match API requirements
   - API returns validation error
   - Terraform displays API error message
   - User corrects the function_id format

3. **FunctionId Not Found**: If function_id is provided but doesn't exist (read scenario)
   - API returns "Function not found" error
   - Terraform marks resource as deleted

## Backward Compatibility

### Compatibility Matrix

| Scenario | Impact | Mitigation |
|----------|--------|------------|
| Existing configs without function_id | ✓ No impact | function_id is Optional, old configs work unchanged |
| Existing resources in state | ✓ No impact | function_id is already Computed, populated from API |
| Provider upgrade | ✓ Transparent | No user action required |
| Import existing resources | ✓ No impact | function_id is read from API response |

### Migration Path

No migration is needed. Users can adopt the new parameter incrementally:
1. Upgrade provider version
2. Optionally add `function_id` to new resources
3. Existing resources continue working as before

No breaking changes are introduced. The change is purely additive (making a Computed field also Optional).

## Performance Considerations

- **API Calls**: No additional API calls (same as current behavior)
- **State Size**: No increase (function_id already existed)
- **Validation**: Negligible overhead from optional parameter check

## Security Considerations

- **FunctionId**: Is a resource identifier, not sensitive data
- No additional security considerations
