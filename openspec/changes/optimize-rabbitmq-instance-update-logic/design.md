## Context

当前的 RabbitMQ 实例 Update 逻辑存在以下限制：
1. 仅支持 `cluster_name` 和 `resource_tags` 两个字段的更新
2. 将 `zone_ids`、`vpc_id`、`subnet_id`、`node_spec`、`node_num`、`storage_size`、`enable_create_default_ha_mirror_queue`、`auto_renew_flag`、`time_span`、`pay_mode`、`cluster_version`、`band_width`、`enable_public_access` 等所有字段标记为不可变
3. 更新操作仅调用一次 API，没有重试机制和状态检查

腾讯云 TDMQ API 实际上支持部分实例属性的动态更新，当前的实现没有充分利用 API 的能力。

## Goals / Non-Goals

### Goals
- 支持 `node_spec`、`node_num`、`storage_size`、`auto_renew_flag` 等字段的动态更新
- 改进 Update 操作的错误处理和重试机制
- 添加更新后的状态同步检查，确保更新成功
- 保持向后兼容性，不破坏现有资源

### Non-Goals
- 不支持 `zone_ids`、`vpc_id`、`subnet_id` 等核心网络配置的更新（这些字段需要重建实例）
- 不支持 `cluster_version` 的变更（版本升级需要专用 API）

## Decisions

### 1. 动态更新字段识别
根据腾讯云 API 文档和实际测试，以下字段支持动态更新：
- `node_spec`: 节点规格调整（需要集群处于运行状态）
- `node_num`: 节点数量调整（需要集群处于运行状态）
- `storage_size`: 存储大小调整（需要集群处于运行状态）
- `auto_renew_flag`: 自动续费标志（可随时更新）
- `band_width`: 公网带宽调整（需要已开启公网访问）
- `enable_public_access`: 公网访问开关（需要集群处于运行状态）

**决策**: 在 Update 函数中移除上述字段的不可变限制，允许用户通过 Terraform 更新这些属性。

**理由**: 这些字段在腾讯云控制台均支持在线修改，没有技术限制阻止通过 API 更新。移除限制可以为用户提供更灵活的管理能力。

### 2. 多阶段更新策略
不同字段的更新可能需要不同的 API 调用或操作顺序：
- 规格/存储/节点数调整：需要等待集群稳定状态
- 公网访问开关：可能需要先调整带宽再开启/关闭访问

**决策**: 实现多阶段更新逻辑，按照依赖顺序依次更新字段。

**理由**: 确保更新的原子性和可靠性，避免因字段依赖导致的更新失败。

### 3. 更新状态验证
当前实现仅执行 API 调用，不检查更新是否真正成功。

**决策**: 在 Update 操作完成后，调用 Read 函数验证所有更新字段是否已正确应用。如果验证失败，返回错误并记录日志。

**理由**: 提高可靠性和用户体验，避免状态不一致问题。

### 4. 错误处理和重试
当前实现没有针对 Update 操作的错误重试机制。

**决策**: 为所有 API 调用添加标准的重试逻辑（使用 `resource.Retry`），并记录详细的调试日志。

**理由**: 云 API 可能因临时网络问题或服务负载而失败，重试机制可以提高成功率。

### 5. 向后兼容性
需要确保更新后的代码不会影响现有资源的状态。

**决策**: 所有新增的更新字段都是可选的，未使用这些字段的资源不受影响。保持现有 `cluster_name` 和 `resource_tags` 的更新逻辑不变。

**理由**: 向后兼容性是 Terraform Provider 的重要原则，避免破坏用户现有资源。

## Risks / Trade-offs

### Risk 1: API 兼容性风险
某些字段可能在某些特殊情况下不支持更新（例如实例状态不正常、配额限制等）。

**缓解措施**:
- 在 Update 函数中添加实例状态检查，确保集群处于运行状态
- 捕获并处理 API 返回的错误，提供清晰的错误信息
- 在文档中明确说明更新的前提条件和限制

### Risk 2: 更新顺序依赖
多个字段的更新可能存在依赖关系，错误的顺序可能导致更新失败。

**缓解措施**:
- 定义清晰的更新顺序规则（例如：先调整规格，再调整存储）
- 前置检查依赖字段的当前状态
- 使用幂等操作确保即使多次更新也不会产生副作用

### Trade-off 1: 更新速度 vs 可靠性
添加状态验证会增加 Update 操作的耗时。

**理由**: 验证机制确保状态一致性，对于生产环境至关重要。用户可以容忍稍慢的更新速度，但不能接受状态不一致。

### Trade-off 2: 灵活性 vs 复杂性
支持更多字段的更新增加了代码的复杂度和维护成本。

**理由**: 用户对灵活性的需求远大于维护成本的增加。通过良好的代码结构和测试，可以控制复杂性。

## Implementation Plan

1. **移除不可变字段限制**
   - 从 `immutableArgs` 数组中移除 `node_spec`、`node_num`、`storage_size`、`auto_renew_flag`、`band_width`、`enable_public_access`
   - 保留 `zone_ids`、`vpc_id`、`subnet_id`、`cluster_version` 等真正不可变的字段

2. **实现多字段更新逻辑**
   - 为每个可更新字段添加独立的更新分支
   - 构造对应的 API 请求参数
   - 按照 API 文档要求的顺序调用更新接口

3. **添加状态检查**
   - 在更新前检查实例状态，确保处于允许更新的状态
   - 更新后调用 Read 函数验证所有字段是否已正确更新
   - 如果验证失败，返回详细错误信息

4. **改进错误处理**
   - 为所有 API 调用添加重试逻辑
   - 记录详细的调试日志，包括请求和响应内容
   - 提供清晰的错误信息，帮助用户理解问题原因

5. **更新文档和示例**
   - 更新 `tdmq_rabbitmq_vip_instance.html.markdown` 文档
   - 标注哪些字段支持更新、哪些不支持
   - 提供更新操作的示例代码

## Testing Strategy

1. **单元测试**
   - 测试 Update 函数的各种场景：单个字段更新、多字段更新、错误场景

2. **集成测试**
   - 使用真实的 Tencent Cloud 账户测试更新功能
   - 测试不同字段组合的更新
   - 验证更新后状态的一致性

3. **回归测试**
   - 确保现有资源的更新功能不受影响
   - 验证向后兼容性
