## Context

当前 RabbitMQ VIP 实例的 update 逻辑在 `resource_tc_tdmq_rabbitmq_vip_instance.go` 中实现，存在过度限制的问题。代码中定义了一个 `immutableArgs` 列表，包含了大量关键字段（node_spec、node_num、storage_size、band_width、enable_public_access 等），这些字段在 update 操作中被禁止修改。

这种限制可能源于早期 API 能力的限制，但随着腾讯云 TDMQ RabbitMQ 服务的演进，很多字段现在已经可以通过 API 进行修改。当前的实现导致用户无法进行常见的运维操作，如：
- 升级实例规格以应对性能需求
- 增加节点数以提高可用性
- 扩大存储容量以应对数据增长
- 调整公网带宽以优化网络性能
- 开启或关闭公网访问以控制安全边界

## Goals / Non-Goals

**Goals:**
- 解除对可修改字段的限制，允许用户通过 update 操作修改 node_spec、node_num、storage_size、band_width、enable_public_access 等字段
- 优化 update 逻辑，提供更友好的错误提示和状态反馈
- 确保修改后的 update 逻辑与腾讯云 API 能力保持一致
- 保持向后兼容性，不影响现有用户的使用

**Non-Goals:**
- 修改实例的核心架构（如 VPC、子网等基础网络配置）
- 支持所有字段的修改（某些字段可能仍然受腾讯云 API 限制）
- 修改实例的生命周期管理（如删除、迁移等）

## Decisions

### 1. 字段修改策略
**决策：** 采用渐进式解锁策略，首先验证哪些字段可以通过 ModifyRabbitMQVipInstance API 修改，然后从 immutableArgs 列表中移除这些字段。

**理由：**
- 避免一次性解锁过多字段导致兼容性问题
- 通过 API 文档和实际测试确认哪些字段可以修改
- 降低风险，逐步验证每个可修改字段的行为

**替代方案：** 一次性解锁所有字段，但这可能导致 API 调用失败或产生意外行为，风险较高。

### 2. API 调用方式
**决策：** 对于需要修改的字段，如果 ModifyRabbitMQVipInstance API 支持则直接使用该 API；如果需要使用其他 API（如规格变更 API），则在 update 函数中调用相应的 API。

**理由：**
- 保持代码结构的简洁性和可维护性
- 最大化利用现有 API 能力
- 避免过度复杂的条件逻辑

**替代方案：** 为每个字段单独调用 API，但这会增加代码复杂度和 API 调用次数。

### 3. 错误处理和状态反馈
**决策：** 在 update 函数中增加更详细的错误处理，对于 API 不支持的修改操作，返回明确的错误信息，指导用户了解哪些字段可以修改。

**理由：**
- 提升用户体验，避免用户在尝试修改不可修改字段时产生困惑
- 便于排查问题和调试
- 与 Terraform 的错误处理最佳实践保持一致

**替代方案：** 静默失败或返回通用错误，但这不利于用户理解问题。

### 4. 测试策略
**决策：** 为每个新支持修改的字段添加单元测试和集成测试，验证修改操作的正确性和状态更新。

**理由：**
- 确保功能的正确性和稳定性
- 防止回归问题
- 提供使用示例和参考

**替代方案：** 仅手动测试，但这无法保证代码质量和长期稳定性。

## Risks / Trade-offs

### 1. API 能力不明确的风险
**风险：** 腾讯云 API 的文档可能不完整或更新不及时，某些字段可能不支持修改，导致 update 操作失败。

**缓解措施：**
- 在实际环境中测试每个字段的修改操作
- 监控 API 响应，及时更新 immutableArgs 列表
- 提供清晰的错误信息，告知用户哪些字段可以修改

### 2. 向后兼容性风险
**风险：** 解锁某些字段后，可能导致现有用户的配置行为发生变化（例如，之前禁止修改的字段现在可以修改，可能导致意外的状态变更）。

**缓解措施：**
- 保持默认行为不变，仅允许显式修改
- 在文档中明确说明可修改的字段及其影响
- 在代码中增加版本兼容性检查

### 3. 性能影响
**风险：** 支持更多字段的修改可能导致 update 操作的耗时增加，特别是在需要调用多个 API 的情况下。

**缓解措施：**
- 优化 API 调用逻辑，减少不必要的请求
- 增加 Timeouts 支持，允许用户配置超时时间
- 提供进度反馈，让用户了解 update 操作的状态

### 4. 测试覆盖率不足
**风险：** 新增的 update 逻辑可能无法覆盖所有边缘情况，导致生产环境出现问题。

**缓解措施：**
- 编写全面的单元测试和集成测试
- 在预发布环境中充分测试
- 提供详细的错误日志和监控

## Migration Plan

### 部署步骤

1. **阶段 1：调研和验证**
   - 查阅腾讯云 TDMQ RabbitMQ API 文档，确认哪些字段支持修改
   - 在测试环境中验证每个字段的修改操作
   - 更新 immutableArgs 列表，移除可修改的字段

2. **阶段 2：代码实现**
   - 修改 `resource_tc_tdmq_rabbitmq_vip_instance.go` 中的 update 逻辑
   - 增加错误处理和状态反馈
   - 确保向后兼容性

3. **阶段 3：测试和验证**
   - 编写单元测试和集成测试
   - 在测试环境中验证所有修改操作
   - 检查错误处理和状态更新

4. **阶段 4：文档更新**
   - 更新资源文档，说明可修改的字段
   - 提供使用示例和最佳实践
   - 更新 CHANGELOG

5. **阶段 5：发布和监控**
   - 发布新版本的 Terraform Provider
   - 监控生产环境中的使用情况和错误日志
   - 收集用户反馈，及时修复问题

### 回滚策略

如果出现问题，可以通过以下方式回滚：

1. **代码回滚：** 恢复到修改前的版本，重新发布
2. **文档回滚：** 更新文档，说明当前版本的限制
3. **用户通知：** 通知用户使用旧版本或避免某些操作

## Open Questions

1. **API 能力验证：** 哪些字段实际上可以通过 ModifyRabbitMQVipInstance API 修改？是否需要调用其他 API？

2. **并发修改：** 如果多个字段同时被修改，应该如何处理 API 调用顺序和依赖关系？

3. **状态一致性：** 修改某些字段（如 node_spec）可能导致实例状态变更（如从 Running 变更到 Updating），如何确保状态一致性？

4. **错误恢复：** 如果 update 操作中途失败，如何处理部分修改的情况？是否需要回滚或重试？

5. **性能优化：** 对于需要多次 API 调用的场景，如何优化性能，减少不必要的请求？
